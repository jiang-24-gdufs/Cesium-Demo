# 二维↔三维属性查询联动验证报告

### 结论：**联动不可行** (当前数据条件下)

---

### 一、验证过程

#### 1. 二维 DWG 端数据结构

二维地图服务 `dongchesuo-ditu` 包含 18 个可查询子图层（数据源 `dongchesuo-dwg`），分为四类几何类型：
- **TEXT (_T)**：DWG 文字注记（如标注尺寸、楼层标高）→ `BlockName` 字段存在
- **LINE (_L)**：DWG 线段图元
- **LINE3D (Z_L)**：DWG 三维线段
- **REGION (_R)**：DWG 填充面

**DWG 要素的字段列表** (以 _T 图层为例)：
`SmID, SmUserID, EntityHandle, EntityType, Layer, LayerFroze, LayerLocked, LayerOn, Elevation, Thickness, ColorIndex, LineStyle, BlockName, BlockRotation, LineWidth, X, Y, Z, Radius, StartAngle, EndAngle, Color`

关键观察：
- **没有 `ELEMENTID`、`UNIQUEID`、`CATEGORYID` 等 BIM 语义字段**
- 唯一可能的关联线索是 `BlockName` 字段，其值为：`广州动车所四线检查库_1-1剖视图_去图框__1__dwg-554611-剖面 1`
- 正则 `/_dwg-(\d+)-/` 可从中提取出 `554611`

#### 2. 三维 BIM 端数据结构

三维 S3M 数据服务 `data-dongchesuotest` 包含 15 个数据集（数据源 `dongchesuo`），字段列表 (以 `qiang_dongchesuo` 为例)：
`SMID, SMUSERID, SMMAXZ, SMMINZ, SMINDEXKEY, SMBIMINFO, ELEMENTID, ELEMENTNAME, CATEGORYID, UNIQUEID, 类别, 横截面, IFC_预定义类型, ...`

关键字段：
- `ELEMENTID`：Revit 构件 ID（如 `341012`，对应 `基本墙[341012]`）
- `UNIQUEID`：Revit GUID（如 `e3985d9a-41b0-41f2-9170-738092ce4906-00053414`）
- `SMBIMINFO`：BIM 层级信息（如 `2F-3.850#*#*#墙#*#*#基本墙#*#*#常规 - 200mm#*#*#基本墙[341012]`）

#### 3. 关键验证：BlockName 中的 554611 能否在三维中匹配？

**实际查询结果**：遍历全部 15 个三维数据集，`ELEMENTID = 554611` 在所有数据集中均返回 `totalCount=0`。

### 二、失败根因分析

#### 根因 1：二维 DWG 与三维 BIM 来自不同的数据导出管线

| 维度 | 二维 DWG 剖面图 | 三维 BIM S3M 模型 |
|------|-----------------|-------------------|
| 数据源名 | `dongchesuo-dwg` | `dongchesuo` |
| 数据来源 | Revit 导出 DWG (剖面视图) | Revit 导出 S3M (三维模型) |
| 几何类型 | TEXT/LINE/REGION (2D投影) | GEOMODEL3D (三维体) |
| 坐标系 | 投影坐标 (mm级，范围 ~65000×24000) | WGS84 经纬度 (度级) |
| 要素粒度 | DWG Block/线段/面（CAD图元级） | BIM 构件级 (墙/柱/梁) |
| 业务字段 | EntityHandle, Layer, BlockName | ELEMENTID, UNIQUEID, SMBIMINFO |

#### 根因 2：BlockName 中的 ID 不对应三维构件

二维 _T 图层只有 **1 个要素**（SmID=1），其 BlockName 为：
```
广州动车所四线检查库_1-1剖视图_去图框__1__dwg-554611-剖面 1
```

这个 `554611` **不是一个具体的墙/柱/梁构件的 ElementID**，而是 Revit 中「剖面视图」对象本身的 ElementID。DWG 导出时将整个剖面视图作为一个 Block 插入，BlockName 中编码了视图级别的引用 ID，而非构件级别的 ID。

#### 根因 3：二维 DWG 的 LINE/REGION 图层无任何 BIM 语义字段

除 _T (TEXT) 图层有 BlockName 外，_L (LINE)、Z_L (LINE3D)、_R (REGION) 图层的字段仅为 DWG CAD 属性（EntityHandle, Layer, ColorIndex, LineStyle 等），**完全没有 ELEMENTID、UNIQUEID 等 BIM 关联字段**。

即使用户在二维地图上拾取了一条代表"墙"的线段，该线段的属性中也没有任何信息能反向定位到三维 BIM 中对应的哪面墙。

#### 根因 4：坐标系不可互转

- 二维使用 CRS.Simple（投影坐标，单位 mm）
- 三维使用 WGS84 经纬度
- 两者之间没有已知的投影参数关系，无法通过坐标空间匹配

### 三、当前代码中联动逻辑的状态

代码 `app.js` 中已实现的联动路径及其实际效果：

| 联动方向 | 实现逻辑 | 实际效果 |
|----------|----------|----------|
| **三维→二维** (拾取) | 三维拾取 → 获取 ELEMENTID → 在二维地图放标记 + popup | ✅ 标记可放置，但位置是经纬度转投影坐标，**位置不准确** |
| **二维→三维** (拾取) | 二维拾取 → BoundsQuery → 提取 BlockName → 正则取 ElementID → 查三维数据集 | ❌ **查无结果**（554611 不在三维数据集中） |
| **三维→二维** (高亮) | N/A | ❌ 二维 DWG 无法按 BIM 构件高亮 |
| **二维→三维** (高亮) | `highlightInModelByKey()` | ❌ 无有效 linkKey |

### 四、后续如果需要满足联动的条件

要实现真正的二维↔三维属性联动，需满足以下 **任意一组** 条件：

#### 方案 A：数据层面 — 重新导出 DWG 并注入 BIM 属性 (推荐)

1. **Revit 端**：使用支持 BIM 属性导出的 DWG 导出插件（如 SuperMap BIM 工具），在 DWG 导出时将每个图元的 `ElementID`/`UniqueID` 写入 DWG 扩展属性（ExtendedData/XDATA 或自定义属性字段）
2. **iServer 端**：确保 DWG 数据集的字段表中包含 `ELEMENTID`（或自定义关联字段），与三维 S3M 数据集中的 `ELEMENTID` 一致
3. **前端代码**：当前 `extractLinkKey()` 和 `highlightInModelByKey()` 逻辑已就绪，数据准备好后无需改代码

#### 方案 B：中间映射表

1. 在 iServer 中创建一张映射表（如 `bim_dwg_mapping`），包含字段：`ELEMENTID`（BIM构件）、`DWG_ENTITYHANDLE`（DWG图元 Handle）、`DWG_LAYER`（DWG图层名）
2. 由 Revit/BIM 工具批量生成这张表
3. 前端联动时：二维拾取 → 获取 EntityHandle → 查映射表 → 得到 ELEMENTID → 查三维数据集

#### 方案 C：坐标空间匹配（精度有限）

1. 建立二维投影坐标 → WGS84 经纬度的转换关系（需要至少 3 个控制点）
2. 二维拾取点位 → 转经纬度 → 在三维场景中做空间查询（BoundingSphere/Ray）匹配最近构件
3. **缺点**：精度受控制点质量影响，密集构件区域可能误匹配

#### 方案 D：使用 Revit 原生剖面视图服务替代 DWG

1. 不使用 DWG 导出的二维剖面图
2. 改用 SuperMap iServer 直接发布 Revit 模型的剖面切割视图（如 S3M 剖面服务）
3. 这样二维和三维共享同一份 BIM 数据源，天然具备 ELEMENTID 关联

---

### 五、总结

| 项目 | 状态 |
|------|------|
| 二维查询功能 | ✅ 正常（BoundsQuery 可查到 DWG 要素属性） |
| 三维查询功能 | ✅ 正常（SQL 查询 ELEMENTID/UNIQUEID 均可用） |
| 二维→三维联动 | ❌ **不可行**（DWG 无 BIM 关联字段） |
| 三维→二维联动 | ❌ **不可行**（坐标系无法互转，且二维无构件级高亮） |

**核心阻塞**：二维 DWG 数据和三维 BIM S3M 数据之间缺少共同的构件级关联字段，且坐标系不可互转。这是数据层面的问题，不是代码层面的问题。
