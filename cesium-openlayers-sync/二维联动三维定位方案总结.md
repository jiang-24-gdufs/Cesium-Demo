# 二维联动三维定位方案总结

本文档总结了两个 chat session 中尝试过的所有**二维点击 → 三维构件定位**的方案及其实际效果。

---

## 1. 核心问题

点击二维 MVT feature 后，需要联动飞行到 Cesium（SuperMap）三维场景中对应的 BIM 构件。关键难点在于：

- BIM 3D 数据集的 iServer SQL 查询结果**不返回** `SMSDRIW/E/N/S`（空间范围极值）
- BIM 3D 数据集查询结果**不返回** `geometry`（平面几何）
- BIM 3D 数据集查询结果**不返回** `SMX/SMY`（空间中心坐标）
- SuperMap `S3MTilesLayer` **没有公开** `按 SmID 获取构件包围盒` 的 API
- 唯一可靠返回的空间信息是 `SMMAXZ/SMMINZ`（构件高度范围）

## 3. 失败原因汇总

1. **iServer 数据服务限制**：BIM 3D 数据集（dongchesuo 系列）查询时不返回 SMSDRI*、geometry、SMX/SMY 等空间字段，只有属性字段 + SMMAXZ/SMMINZ
2. **SuperMap S3MTilesLayer API 限制**：没有 `getFeatureBounds(smId)` 或类似的公开方法，`layer.layerBounds` 是整层范围
3. **瓦片树遍历不可靠**：不同 iClient3D 版本的内部结构不同，`_cache`/`_renderEntityMap`/`_batchTable` 格式不确定；目标瓦片可能未加载
4. **二维 MVT feature 有精确几何**：但之前没有把这个信息传递到定位流程中

---

## 4. 当前代码状态

`flyToComponent3D` 保留了两级策略：
- 策略 A（瓦片遍历）— 实际不工作
- 策略 B（图层包围球 fallback）— 精度不足

二维→三维的联动定位**实质上没有正确实现构件级聚焦**。

## 5. GPT5.6Sol 推荐方案及其局限

参考 `二维点击聚焦三维构件实现方案.md`，推荐**两阶段聚焦**：

### 阶段一：可靠粗定位

利用已有信息构造三维包围球：
- 水平范围：**二维 MVT feature 的 geometry extent**（EPSG:3857，通过 `feature.getGeometry().getExtent()` 获取）
- 或 iServer REST 查询返回的 feature geometry bounds
- 垂直范围：iServer 查询返回的 `SMMINZ/SMMAXZ`
- 将以上信息组合为 `Cesium.BoundingSphere.fromPoints` → `flyToBoundingSphere`

### 阶段二：场景精修

第一次飞行完成后，目标瓦片已加载：
- 按 `S3M layer + SmID` 在当前画面采样 `scene.pick` + `scene.pickPosition`
- 命中足够的三维表面点 → 构造更精确的 BoundingSphere → 短距离二次飞行
- 未命中 → 保留第一阶段结果

### ⚠️ 该方案的隐含假设与局限

**该方案默认二维 MVT 和三维 S3M 模型处于相同的空间原点和坐标系统**。具体来说：

- MVT 瓦片的几何坐标 (EPSG:3857 墨卡托) 通过 `ol.proj.toLonLat` 转为经纬度后，直接作为 Cesium 的 `fromDegrees` 输入
- 三维 BIM 模型在 iDesktop 发布时可能有自定义的插入点偏移（`position` 属性）
- 如果 MVT 矢量面是从 BIM 模型的**俯视投影**生成的，它的平面范围与三维构件的真实经纬度范围可能一致
- 但如果 MVT 是独立制作的（不来自同一模型），或者 BIM 模型发布时有局部坐标偏移，二者的坐标**不会对应**

**实际影响**：阶段一的粗定位可能飞到一个完全错误的位置（偏移数百米甚至数公里），此时阶段二的屏幕采样也不可能命中目标构件。

**验证方法**：在代码中添加日志输出二维 extent 转换后的经纬度范围，与三维场景中模型的实际经纬度范围对比，如果存在系统偏移则该方案不适用。

## 6. 基于三维对象 API 的方案分析

### 6.1 SuperMap S3MTilesLayer 公开 API 清单

经查阅官方文档（S3MTilesLayer API），与"获取选中对象空间位置"相关的公开接口：

| API | 功能 | 能否提供构件空间位置 |
|---|---|---|
| `layer.setSelection(ids)` | 设置选中集 | 否（仅选中，不返回空间信息） |
| `layer.getSelection()` | 获取选中集 | 返回 AssociativeArray（ID 集合），无空间信息 |
| `layer.getAttributesById(id)` | 获取构件属性（需 indexedDB / setQueryParameter） | **可能**：如果属性中包含自定义的坐标字段 |
| `layer.getObjsVisible(id)` | 获取对象可见性 | 否 |
| `layer.layerBounds` | 图层范围 (Rectangle) | 整层范围，非构件级 |
| `layer._boundingSphere` | 图层包围球 | 整层范围，非构件级 |
| `scene.pick(windowPosition)` | 拾取 | 返回 { primitive, id }，无空间坐标 |
| `scene.pickPosition(windowPosition)` | 拾取深度缓冲位置 | **是**：返回精确的世界坐标 Cartesian3 |
| `layer.selectedTranslate` | 选中对象的偏移 | 可读写偏移量，但不是位置 |

### 6.2 可行的"基于三维对象"方案

**核心思路**：在 `setSelection([smId])` 之后，不尝试从数据中计算位置，而是利用 Cesium 的渲染管线在**下一帧**从屏幕上"找到"这个被选中的对象。

```
setSelection([smId])  →  等待下一帧渲染  →  逐像素 scene.pick 扫描
  →  命中 target.layer + target.smId 的像素  →  scene.pickPosition 获取世界坐标
  →  从所有命中点构建 BoundingSphere  →  flyToBoundingSphere
```

**可行性判断：偏乐观，但有条件限制**：

✅ **可行的前提**：
- 目标构件已经在当前视锥内**可见**（像素已渲染到屏幕上）
- `scene.pickPositionSupported` 为 true（需要深度缓冲可用）
- 构件没有被其他对象完全遮挡

❌ **不可行的场景**：
- 目标构件**不在当前视野内**（相机还没飞过去，屏幕上根本没有这个构件的像素）
- 目标构件**瓦片未加载**（S3M LOD 调度未请求该 tile）

**结论**：该方案**适合作为阶段二的精修手段**（第一次粗飞行后构件进入视野），但**不能作为阶段一的独立方案**，因为无法定位不在当前视野内的构件。

### 6.3 `getAttributesById` 的定位潜力

如果在 iDesktop 发布模型时为每个构件写入了中心点坐标字段（如 `CENTER_X`, `CENTER_Y`, `CENTER_Z`），`getAttributesById` 就能直接拿到定位信息。

**当前数据的情况**：根据 `feature.fields.md` 样例，属性中只有 `SMMAXZ/SMMINZ`，没有自定义坐标字段。但 `getAttributesById` 的返回结果**可能**与 `doSqlQuery` 不同（来自 S3M 属性文件而非数据服务），值得通过日志验证。

## 8. 定位数据优先级（最终方案）

| 优先级 | 信息来源 | 说明 |
|---:|---|---|
| 1 | 发布时写入的构件三维 bbox/center | 最稳定（需数据生产支持） |

## 9. Session 3 决策：放弃二维联动三维定位

### 9.1 结论

**彻底放弃**二维拾取属性查询联动三维"定位"交互。

经过 3 个 session 的反复验证，所有从数据维度获取构件空间坐标的尝试均失败：

| Session | 尝试的策略 | 结果 |
|---:|---|---|
| 1 | SMSDRI 包围盒、geometry bounds、SMX/SMY、墨卡托投影、S3M 瓦片遍历 | 全部从未生效 |
| 2 | S3M 瓦片缓存遍历（改进版）、图层包围球 fallback | 瓦片遍历从未生效 |
| 3 | iServer hasGeometry=true 重查、targetLayer.lon/lat+fixedHeight、改进瓦片遍历 | 全部从未生效 |

### 9.2 Session 3 新增尝试的失败分析

**策略 A：iServer hasGeometry=true 几何查询**
- `queryParameter.hasGeometry` 默认为 false，尝试设为 true 重查
- BIM 3D 数据集（dongchesuo 系列）即使 hasGeometry=true，返回的 feature.geometry 依然为 null
- 根本原因：BIM 数据集本身不存储平面几何

**策略 B：S3M 瓦片树遍历（改进版）**
- 增加了 `_renderEntityMap`、`_renderEntities`、`_ids`、`_batchTable` 多路径查找
- 实测 `layer._cache` / `layer._tileCache` 均为 undefined
- SuperMap iClient3D 11.2.1 的 S3MTilesLayer 内部结构与假设不符

**策略 C：targetLayer.lon/lat + fixedHeight**
- `targetLayer.lon = 0.0069°`，`targetLayer.lat = 0.0130°`（positionUnits="Degree"）
- 这些值是图层的**偏移量/插入点**，非绝对经纬度
- 定位结果落在赤道附近（0°,0° 偏移量），完全错误

**策略 D：targetLayer._boundingSphere**
- `_boundingSphere.center` === `Cesium.Cartesian3.ZERO`
- `max/min` 均为 `{x:0, y:0, z:0}`
- 无法用于定位
